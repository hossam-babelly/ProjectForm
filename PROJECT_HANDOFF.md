# 📘 دليل المشروع الشامل — للمتابعة في Claude Code

> هذا الملف مرجع تسليم (Handoff) كامل لمشروع **«نموذج طرح دراسة مشروع»**.
> الهدف منه أن يفهم أي مطوّر (أو Claude Code) المشروع بالكامل دون قراءة كل الكود سطراً سطراً:
> الغاية، البنية، خريطة الكود، نظام الألوان والتنسيقات البصرية، آلية الاستخدام، والنقاط التي يُبنى عليها العمل القادم.
>
> **النسخة:** v3 (بعد جولتَي التعديل) · **اللغة:** عربي RTL · **آخر تحديث للوثيقة:** يونيو 2026.

---

## 1) الغاية من المشروع

نظام ويب متكامل (Frontend + Backend) لمؤسسة **«المجد التنموية»**، يستقبل بيانات **دراسة جدوى مشروع** عبر فورم عربي متعدّد الخطوات، ثم يُولّد تلقائياً ملفّين احترافيّين:

- **ملف Word (.docx)** — دراسة جدوى منسّقة RTL بترويسة المؤسسة، جداول مالية بعرض الصفحة، هيكل تنظيمي ملوّن، ولوحة خلاصة مالية بصرية.
- **ملف Excel (.xlsx)** — مصنّف من 12 ورقة مترابطة بمعادلات حيّة (تعديل أي مُدخل يُعيد حساب كل المخرجات).

عند الإرسال: يُحفظ الطلب، يُولَّد الملفّان، **يُحمَّلان فوراً** في المتصفّح، ويُرسَلان **في الخلفية** بالبريد عبر **Brevo HTTP API**. وهناك **لوحة إدارة** لعرض/تنزيل/تعديل/حذف الطلبات.

**الجمهور:** مقدّمو طلبات دعم المشاريع + موظّفو المؤسسة الذين يستلمون الدراسات جاهزة للطباعة/الأرشفة.

## 3) خريطة الكود (Code Map)

> الأرقام تقريبية وقد تنزاح بعد التعديل؛ اعتمد على **أسماء الدوال** و**النصوص المرجعية (anchors)** للبحث.

### 3.1 `server.js` — نظرة عامة على الأقسام

| القسم | المحتوى |
|---|---|
| الرأس (`require` + ثوابت) | `PORT`، `DATA_DIR`، `MONTHS`، `genId()`، `loadIndex()/saveIndex()`، قراءة `letterhead.png` |
| `generateExcel(data)` | يبني المصنّف بالكامل (12 ورقة) ويُرجع Buffer |
| `generateWord(data)` | يبني مستند docx، ثم يمرّره على خطّ الحقن، ويُرجع Buffer |
| دوال الهيكل التنظيمي | `buildOrgTree()` · `buildOrgChartParagraphXml()` · `injectOrgChart()` |
| دوال اللوحة المالية | `buildFinanceDashXml()` + مساعداتها (`fwRect`/`fwRing`/`fwBar`/`fwTxt`/`fwAnchor`) · `injectFinance()` |
| `finalizeDocx()` | يصلح علاقة `fontTable.xml` كي لا يُظهر Word «محتوى غير قابل للقراءة» |
| البريد | `sendViaBrevo()` · `sendEmail()` |
| المسارات (Routes) | `/api/submit` وغيرها (انظر §10) |

### 3.2 مولّد Word — التفاصيل

- **ثوابت التصميم:** كائن الألوان `C` (انظر §9)، `PAGE` (A4 أفقي، هوامش)، `TW` (عرض الجدول = 15388 DXA)، `FONT='Sakkal Majalla'`، و`COLW` (عرض أعمدة كل جدول بدقّة مستخرجة من النموذج الرسمي).
- **مساعدات الخلايا/الجداول:** `C_(text, opts)` خليّة (RTL، `bidirectional:true`)، `secHdr()` عنوان قسم، `colHdr()` رأس أعمدة، `totRow()` صفّ إجمالي، `lblValTable()` جدول «تسمية/قيمة»، `T_()` تغليف جدول، `page()/mkSec()` بناء قسم/صفحة، `norm()` معايرة عرض الأعمدة، `fM()` تنسيق مبلغ بالدولار و`fN()` تنسيق رقم.
- **بناء الصفحات:** تُبنى الأقسام في كائن `S` (`S.applicant`, `S.project`, ...) ثم تُرتَّب في المصفوفة النهائية:
  ```js
  [S.applicant, S.project, S.mkt, S.founding, S.products, S.revenue,
   S.ops, S.hr, S.org, S.fixed, S.dep, S.summary]
  ```
- **خطّ الحقن بعد البناء (placeholder → XML):**
  - صفحة الهيكل (`S.org`) تحوي عنصراً نائباً `[[ORGCHART]]` → يستبدله `injectOrgChart()` بجدول Word أصلي ملوّن.
  - صفحة الخلاصة (`S.summary`) تحوي `[[FINANCE]]` → يستبدله `injectFinance()` بأشكال DrawingML (بطاقات + حلقات + أشرطة).
  - أخيراً `finalizeDocx()`.

### 3.3 مولّد Excel — التفاصيل

- **12 ورقة** كلّها RTL (`views:[{rightToLeft:true}]`) بنفس ترتيب Word.
- **الترابط بين الأوراق:** كائن `refs` يخزّن مراجع خلايا (مثل `refs.revAnnual`, `refs.opsAnnual`, `refs.fixedAnnual`, `refs.founding`, `refs.depTotal`, `refs.hrAnnual`, `refs.mktMonthly`)؛ وورقة الخلاصة تستهلكها بمعادلات حيّة.
- **مساعدات:** `dcell()` خليّة بيانات، `colHeaders()`، `totalRow()`، `band()` شريط عنوان، `kv()` صفّ «تسمية/قيمة» (يدمج B:C)، و`kvPct()` صفّ نسبة (رقم في B + **دائرة أيقونة** في C، **بلا دمج** — انظر §8).
- **التنسيق الشرطي في ورقة الخلاصة:** أيقونات `5Quarters` (دوائر نِسَب) على `C11/C12`، وأشرطة بيانات `dataBar` على `B15:B17`.
- **اصطلاح الألوان للمُدخلات/الروابط/المعادلات:** أزرق = مُدخل، أخضر = رابط بين الأوراق، أسود = معادلة محسوبة (موضّح أسفل كل ورقة).

### 3.4 الواجهة `index.html`

- `dir="rtl"`، خط Cairo. `TOTAL_STEPS = 11`. مصفوفة `labels` تحوي عناوين الخطوات الإحدى عشرة.
- `updateSummary()` تُعيد حساب كل المؤشّرات حيّاً عند أي تغيير وتملأ: بطاقة الملخّص العلوية (`sum_*`)، وصفحة الخلاصة المالية بالخطوة 11 (`fin_*`)، وحلقتَي الهامش/العائد (`setGauge`).
- `submitForm()` تجمع كامل الحمولة (§4) وترسلها إلى `/api/submit`، ثم تُنزّل الملفّين عبر `downloadBoth()`.

### 3.5 لوحة الإدارة `admin.html`

صفحة مستقلّة (خط Cairo، ألوانها الخاصّة §9) تستدعي `/api/submissions`، وتعرض بطاقات للطلبات مع أزرار: تنزيل Excel/Word، تعديل، حذف.

## 4) نموذج البيانات (Payload schema)

عند الإرسال يبني الفورم كائناً يُرسَل إلى `POST /api/submit`:

```jsonc
{
  // مُقدّم الطلب
  "applicantName", "applicantPhone", "applicantBirth",
  "applicantResidence", "applicantJob", "applicantExpYears", "applicantExperience",
  // المشروع
  "projectTitle", "projectIdea",
  "targetSegments", "placeType", "placeStatus", "projectAddress",
  // التسويق
  "marketingPlan", "marketingCost", "salesChannels",
  // المنتجات (كل منتج: اسم + وحدة + مكوّنات + منافسون)
  "products": [ { "name", "unit", "components":[...], "competitors":[...] } ],
  // جداول مالية
  "revenueData": { "<productId>": [ {qty,price,total} × 12 شهر ] },
  "opsData":     { "<productId>": { "sub_<m>":…, "<comp>_<m>":… } },
  "fixedRows":   [ { cat, bayan, notes, total, … } ],   // تكاليف ثابتة شهرية
  "hrRows":      [ { posId, position, type, count, salary, … } ], // الموارد البشرية
  "depRows":     [ { cat, bayan, pct, perUnit, qty, total, notes } ], // الاهتلاك
  // الخلاصة المحسوبة مسبقاً في المتصفّح (تُستخدم في لوحة Word + بطاقة الإدارة)
  "summary": {
    "foundingTotal", "revenueAnnual", "opsAnnual",
    "fixedMonthly", "fixedAnnual", "marketingMonthly", "marketingAnnual",
    "depreciation", "netProfit", "employees",
    "paybackYears", "netMarginPct", "roiPct"
  },
  "submittedAt"
}
```

> ملاحظة: قيم `summary` تُحسب في المتصفّح للعرض، لكن **Excel يُعيد حسابها بمعادلاته الخاصّة** (مصدر الحقيقة في الملف هو المعادلات، لا قيم summary).

## 7) اللوحة المالية في Word (صفحة الخلاصة)

تُحقَن كأشكال DrawingML مرساة بإحداثيات EMU عبر `buildFinanceDashXml(summary)`:

- **5 بطاقات** علوية بالترتيب (يساراً→يميناً): الإيرادات · التشغيلية · الثابتة · **الاهتلاك** · التأسيسية — أي من اليمين: التأسيسية ثم **الاهتلاك (ثاني بطاقة)** ثم الثابتة ثم التشغيلية ثم الإيرادات.
- **بطاقة صافي الربح** بعرض كامل (كحلية، الرقم برتقالي).
- **بطاقة فترة الاسترداد** + **حلقتان (Rings)** للهامش والعائد (نصوصها داخل الدوائر بحجم ثابت لا يكبر).
- **3 أشرطة مقارنة** (إيرادات/تكاليف/ربح).

> المساعدات: `fwRect` مستطيل/بطاقة · `fwRing` حلقة نسبة · `fwBar` شريط · `fwTxt` نص داخل شكل · `fwAnchor` تثبيت الشكل على الصفحة. الألوان في كائن `FW`.

## 8) دوائر النِسَب في Excel (آليّة العمل بعد الإصلاح)

في ورقة الخلاصة، صفّا **هامش الربح الصافي** و**العائد على الاستثمار** يستخدمان مجموعة أيقونات **`5Quarters`** (دائرة تمتلئ على 5 مستويات):

- الرقم في الخليّة **B**؛ الدائرة (أيقونة فقط، `showValue:false`) في الخليّة **C** المجاورة — **بلا دمج** كي لا يتداخل الرقم مع الدائرة على ورقة RTL.
- العتبات `num`: `0 / 0.2 / 0.4 / 0.6 / 0.8` → فارغة <20% · ربع · نصف · ثلاثة أرباع · ممتلئة ≥80%. القيم السالبة (خسارة) = دائرة فارغة (سلوك صحيح).
- ملاحظة عرض: **LibreOffice لا يرسم أيقونات التنسيق الشرطي على أوراق RTL**، لكنّها تظهر صحيحة في **Microsoft Excel**.

## 9) آلية الاستخدام و واجهة الـ API

**تدفّق المستخدم:** يملأ الفورم (11 خطوة) → يراجع الخلاصة المالية الحيّة → يضغط «إرسال» → يُحفظ الطلب ويُولَّد الملفّان و**يُنزَّلان فوراً** → يُرسَل البريد في الخلفية.

| الطريقة | المسار | الوظيفة |
|---|---|---|
| `POST` | `/api/submit` | حفظ الطلب + توليد Word/Excel + ردّ فوري + بريد بالخلفية |
| `GET` | `/api/submissions` | قائمة الطلبات |
| `GET` | `/api/submissions/:id` | تفاصيل طلب |
| `PUT` | `/api/submissions/:id` | تعديل طلب وإعادة توليد ملفّاته |
| `DELETE` | `/api/submissions/:id` | حذف طلب |
| `GET` | `/api/download/:id/word` | تنزيل Word |
| `GET` | `/api/download/:id/excel` | تنزيل Excel |
| `GET` | `/api/email-test` | تشخيص إعدادات البريد (Brevo) |

- الفورم: `/` · لوحة الإدارة: `/admin.html`

## 10) البريد (Brevo) ومتغيّرات البيئة

الإرسال عبر `sendViaBrevo()` إلى `https://api.brevo.com/v3/smtp/email` (HTTP API، يتجاوز حجب SMTP على Render). إن لم يُضبط المفتاح يتخطّى النظام البريد ويُكمل عمله.

| المتغيّر | إلزامي؟ | الوصف |
|---|---|---|
| `BREVO_API_KEY` | للبريد فقط | مفتاح Brevo API |
| `SENDER_EMAIL` | للبريد فقط | إيميل المُرسِل (**مُفعَّل كمُرسِل في Brevo**) — يقبل `EMAIL_USER` كبديل |
| `RECEIVER_EMAIL` | اختياري | المستلِم (افتراضياً = المُرسِل) |
| `CC_EMAIL` | اختياري | نسخة كربونية (يقبل قائمة مفصولة بفواصل) |
| `PORT` | اختياري | المنفذ (افتراضي 3000؛ Render يضبطه تلقائياً) |

**للتشخيص:** افتح `/api/email-test` — يخبرك إن كان المفتاح/المُرسِل مضبوطاً ويحاول إرسال إيميل تجريبي.

## 11) النشر على Render

1. ارفع الملفات على GitHub عبر **Upload files** (اسحب الملفات المُستخرَجة من الـ Zip — **لا** تحفظها من صفحة عرض GitHub، تفادياً لتلف `package.json`). تأكّد من رفع مجلّد `assets/`.
2. أنشئ **Web Service** واربطه بالمستودع. Build: `npm install` · Start: `npm start` (أو `node server.js`).
3. اضبط متغيّرات البيئة (§10) ثم Deploy.

> ⚠️ **تنبيه التخزين:** على خطة Render المجانية يكون نظام الملفّات **مؤقّتاً** — مجلّد `data/` (الطلبات والملفّات المولّدة) **يُمسح مع كل إعادة نشر/إعادة تشغيل**. للحفظ الدائم استخدم **Render Disk** أو **قاعدة بيانات** (نقطة عمل مستقبلية مهمّة — انظر §13).

## 12) التطوير والاختبار محلياً

```bash
npm install
npm start            # http://localhost:3000   (الفورم)
                     # http://localhost:3000/admin.html  (الإدارة)
```

**للتحقّق البصري من Word/Excel دون متصفّح** (مفيد جداً عند تعديل المولّدات): أرسِل حمولة تجريبية إلى `/api/submit`، ثم حوّل الناتج إلى صور بـ LibreOffice:

```bash
soffice --headless --convert-to pdf data/<id>.docx   # أو .xlsx
pdftoppm -png -r 100 <id>.pdf page                    # عايِن الصفحات
```

> تذكّر: **LibreOffice يعكس بعض نصوص RTL ولا يرسم أيقونات Excel على أوراق RTL** — هذه **قيود عرض في LibreOffice فقط**، والملفّات صحيحة في Microsoft Office. تحقّق دائماً أنّ `word/document.xml` سليم (well-formed) وأنّ عدد الأوراق = 12.

## 13) التعديلات الأخيرة (سجل التغيير)

**الجولة 1 — إظهار إجماليات سنوية:**
- Word: إضافة شريط **«إجمالي الإيرادات السنوية»** أسفل جدول الإيرادات، و**«إجمالي التكاليف التشغيلية السنوية»** أسفل جدول التشغيلية (Excel كان يحويهما أصلاً في صفّ O10/O13).
- إضافة بطاقة **«الاهتلاك السنوي»** إلى لوحة Word المالية وإلى صفحة الخلاصة في الفورم (Excel كان يحويه أصلاً في الخلاصة).

**الجولة 2 — تحسينات بصرية:**
1. **موضع `$`** في صفّ الإجمالي (Word — الإيرادات والتشغيلية): إضافة علامة اتجاه `\u200F` (RLM) في بداية الخلية ليظهر `$` يسار الرقم (مطابقاً للشريط) في MS Word.
2. **صفحة الخلاصة (Word):** نقل بطاقة **الاهتلاك** لتصير ثاني بطاقة من اليمين، وتكبير الخطوط في كامل الصفحة **عدا النصوص داخل الدوائر**.
3. **دوائر النِسَب (Excel):** فصل الدائرة في خليّة C مستقلّة (`kvPct`، أيقونة فقط) بجانب الرقم في B — لإصلاح التداخل على ورقة RTL.
4. **الفورم:** البطاقات الخمس على سطر واحد (`repeat(5,1fr)`) بنفس ترتيب لوحة Word.

## 14) مشاكل وملاحظات معروفة

- **عرض LibreOffice:** عكس نصوص RTL وعدم رسم أيقونات Excel على أوراق RTL — **عرض فقط**، صحيحٌ في MS Office. لا تعتمد على معاينة LibreOffice للحكم على موضع `$` أو الأيقونات.
- **زوال `data/` على Render المجاني:** أهمّ قيد بنيوي للإنتاج (يلزم Disk/قاعدة بيانات للحفظ الدائم).
- **`README.md`** كان قديماً (Gmail/SmartArt/8 خطوات) وتمّ تحديثه ليطابق الواقع (Brevo/جدول أصلي/11 خطوة).
- أغلب الملاحظات القائمة عند المالك **شكليّة (إظهارات)** تُعالَج في مولّد Word/Excel أو في CSS الفورم.

## 15) نقاط الدخول للعمل القادم (Entry Points)

> الخطط المُعلَنة: إضافة ميزات للفورم وللملفّات، وتحسين آلية الاستخدام، ومعالجة ملاحظات شكلية.

| تريد أن… | ابدأ من |
|---|---|
| تعدّل تنسيق/ألوان جدول Word معيّن | كائن `C`، `COLW`، ودالة قسمه `S.<اسم>` في `generateWord` |
| تضيف/تعدّل صفحة Word | أنشئ `S.<جديد>` ثم أدرجه في مصفوفة الترتيب النهائي |
| تعدّل اللوحة المالية لـ Word | `buildFinanceDashXml()` + مساعدات `fw*` + كائن `FW` |
| تعدّل الهيكل التنظيمي | `buildOrgTree()` / `buildOrgChartParagraphXml()` |
| تعدّل ورقة Excel أو معادلاتها | القسم المعنيّ في `generateExcel` + كائن `refs` (احذر كسر الترابط) |
| تعدّل دوائر/أشرطة Excel | `kvPct()` (أيقونات) أو كتلة `dataBar` في ورقة الخلاصة |
| تضيف حقلاً/خطوة للفورم | عدّل HTML الخطوة + `TOTAL_STEPS`/`labels` + الحمولة في `submitForm()` + استهلاكه في `server.js` |
| تعدّل الملخّص الحيّ | `updateSummary()` و`setGauge()` |
| تحسّن لوحة الإدارة | `admin.html` (بطاقات + استدعاءات API) |
| تجعل التخزين دائماً | استبدل قراءة/كتابة `data/*.json` بقاعدة بيانات، أو فعّل Render Disk |
| تشخّص البريد | `/api/email-test` + `sendViaBrevo()` |

**قواعد ذهبية عند التعديل:**
- لا تكسر الترابط بين أوراق Excel (`refs`) — أي إزاحة صفوف قد تُفسد المعادلات؛ تحقّق بإعادة الحساب.
- حافظ على ترتيب الصفحات/الأوراق الموحّد (12) ما لم يُطلب خلافه.
- بعد أي تعديل على المولّدات: ولّد ملفاً فعلياً وعايِنه (§12)، وتأكّد أنّ `document.xml` سليم وأنّ عدد الأوراق 12.
- لا تُدخِل قيم البريد/المفاتيح في الكود — استخدم متغيّرات البيئة.
- عند تغليف الـ Zip للنشر: استبعِد `node_modules/` وملفّات `data/` المولّدة (أبقِ `data/.gitkeep`).
